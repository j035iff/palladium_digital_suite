# Product Vision: Palladium Digital Suite

📌 Core Project Philosophy
Character Nexus is a multi-genre, data-driven tabletop RPG character manager and automation engine designed explicitly to handle the complex, cross-compatible rule ecosystems of the Palladium Megaverse. The core objective is providing absolute rules enforcement and automated calculations while shielding the user from the dense layout complexity of multi-column print materials. 
In short, the app is a "sophisticated calculator". Its explicit purpose is that is strictly a tool for taking the friction out of playing at a table. It is NOT a video game that plays Palladium for you.

🏛️ Pillars of Architecture
1. Setting-First Orchestration (Genre Gating)
The application treats the selected Setting or Genre as the top-level orchestration filter. Rather than running a monolithic rules dictionary, the app operates as a modular framework where selecting a genre loads a scoped Genre Manifest File. This manifest dictates exactly which skills, races, powers, and calculation rules are loaded into system memory.

2. Managed Radical Visibility (Creation vs. Execution)
To maintain user interface clarity without violating structural rules transparency, the app splits dataset visibility into two distinct lifecycle phases:

The Assembly Phase (Character Creation): Absolute Exclusion rules apply. Any trait, skill, or magic discipline not explicitly whitelisted by the active hostGenreId is completely hidden from drop-downs to prevent cognitive clutter.

The Maintenance Phase (Active Play & Leveling): Stark Visual Lockout applies. Universal datasets (such as global skills or psionics arrays) display all entries, but items illegal to the current genre are rendered with a specialized, dark-contrast grey-out design layer. Hovering reveals an explicit global tooltip: "Not available in <current genre>".

3. Centralized Pipeline Transformation (The Middleware Engine)
Character data records remain entirely clean and reflective of their native origins. All environmental adaptation math, multi-genre parameter modifications (such as S.D.C. to M.D.C. tracking matrices), and setting shifts are processed through a isolated Centralized Middleware Layer.

Components are entirely presentation-focused; they never execute environment calculations. They receive a clean, pre-translated payload emitted dynamically by the engine gate.

4. Scalable Multi-Tenant Session Sync (The GM Hook)
The system architecture isolates client state from environment state to future-proof network execution. By anchoring validation rules strictly to a dynamic environment pointer, a standalone Game Master Mode can seamlessly instantiate independent table rooms, forcing connected player payloads to evaluate through the room's host setting auto-magically upon sync.

💾 Core Structural State Schema
Every character file generated by the engine must register and maintain two distinct string tokens at its root-level file state:

interface CharacterRootState {
  id: string;
  name: string;

  readonly creationGenreId: string;

  hostGenreId: string;

}



## Core Pillars
1. **Mechanical Integrity:** Exact adherence to Palladium percentile systems and S.D.C./M.D.C. scaling.
2. **Megaversal Bridge:** Architecture must be setting-agnostic (Fantasy, Rifts, Nightbane).
3. **Speed Over Spectacle:** Two-tap access to primary combat stats.
4. **Total GM Agency:** All values must be overrideable.
5. **Physical Dice Priority:** Frictionless manual entry for real dice results.
6. **Visual Continuity:** Anchored UI elements; navigation stays consistent.
7. **Intuitive Depth:** Surface results first, deep math via tap-to-expand.
8. **Radical Visibility:** NEVER hide restricted options. Gray them out and explain why.
9. **Unified Path:** Facade, Morphus, and dual-form variants are **modes on one pipeline**—not parallel implementations. Each feature domain gets a single math stack, row assembler, tooltip/diff layer, and UI renderer. Extend the shared builder when adding stats or rows; never fork per-stat or per-form code paths.
